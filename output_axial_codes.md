## Axial‑code map  
**Format:** **Condition → Action / Interaction → Consequence**  

The 222 open‑code excerpts were clustered into 12 axial codes that capture the major “story lines” running through the interviews.  Within each axial code the most frequent *conditions* (situational or contextual triggers) are linked to the *actions / interactions* that participants or instructors carried out, and the *consequences* that followed (learning outcomes, affective reactions, or downstream problems).

---

### 1️⃣ Instructional Design & Course Structure  

| Condition | Action / Interaction | Consequence |
|-----------|----------------------|-------------|
| **Limited class time / end‑of‑semester pressure** (Chunks 6, 54, 60, 115) | Compress content, skip deep theory, use “lab‑as‑lecture” (hands‑on coding, in‑class transcription) | Students feel rushed, surface‑level understanding, but appreciate the *practical* exposure. |
| **Introductory‑level exposure only** (Chunks 1, 5, 12, 39) | Rely on PowerPoint, textbook excerpts, “starter slides”, minimal primary‑text reading | Students develop a *neutral* or *superficial* view of qualitative methods; many request deeper literature. |
| **Explicit sequencing (intro → advanced)** (Chunks 14, 45, 67, 170) | Build on prior methodology courses, scaffold with progressive activities (observation → coding → theming) | Learners report smoother transition, higher confidence in later modules. |
| **Pedagogical “refusal” / critical stance** (Chunks 127‑156) | Deliberately withhold prescriptive steps, ask students to question norms, embed refusal language | Students become more reflexive about power, but some feel lost without clear guidance. |
| **Hybrid / online‑vs‑in‑person shift** (Chunks 88‑93, 215‑218) | Switch between Zoom, recorded lectures, face‑to‑face labs; provide recordings, transcripts | Mixed affect: renewed energy in‑person, fatigue and disengagement in Zoom; technical glitches sometimes impede learning. |

### 2️⃣ Prior Knowledge & Student Background  

| Condition | Action / Interaction | Consequence |
|-----------|----------------------|-------------|
| **Quantitative‑heavy backgrounds** (Chunks 28, 62, 94, 194) | Emphasise differences (qualitative = “soft”, quantitative = “hard”), use analogies, ask students to “think like a researcher” | Initial anxiety, perceived “mismatch”, but eventual appreciation of complementary lenses. |
| **Limited exposure to qualitative literature** (Chunks 5, 13, 24, 48) | Provide PowerPoint “cheat‑sheets”, recommend classic texts (Strauss & Glaser, Merriam & Teasdale) | Short‑term clarity; long‑term desire for updated, concrete examples. |
| **Diverse linguistic / cultural backgrounds** (Chunks 65‑68, 71‑73, 151‑154) | Use multicultural examples (Jewish film, Chinese sweatshop, Van Gogh copies) and encourage personal narratives | Heightened engagement, sense of belonging; occasional confusion when cultural references are unexplained. |
| **Low confidence in software / library resources** (Chunks 26, 212, 215‑218) | Offer librarian tutorials, free software licences, step‑by‑step demos | Reduced fear of “tech”, but many still prefer manual/highlighter methods. |

### 3️⃣ Coding Practices & Perceptions  

| Condition | Action / Interaction | Consequence |
|-----------|----------------------|-------------|
| **Uncertainty about code usefulness / hierarchy** (Chunks 4, 9, 16, 46, 75) | Discuss code‑to‑category mapping, show examples, allow “one‑off” codes, colour‑coding on paper | Students gain a *working* sense of coding but remain uneasy about when a code is “important”. |
| **Preference for manual / paper‑based coding** (Chunks 7, 20, 21, 30, 41) | Provide worksheets, coloured pens, dual‑sheet set‑ups, encourage highlighting | Faster for many, perceived as more “authentic”; software seen as extra overhead. |
| **Encounter with preset or a‑priori coding schemes** (Chunks 29, 42, 43, 44, 129‑131) | Offer a theme list, four‑Ws framework, or let groups generate their own codes | Mixed outcomes: some feel guided, others feel constrained; flexibility is valued when optional. |
| **Software expectations vs reality** (Chunks 47, 195, 197) | Demonstrate coding software, clarify it is a *tool* not a replacement | Initial disappointment, later acceptance that software aids organization but does not replace researcher judgment. |
| **Linking coding to theory (grounded theory, discourse analysis, etc.)** (Chunks 13, 56, 77, 98, 210) | Introduce multiple analytic lenses, compare frameworks, ask students to re‑code same text | Broadens methodological repertoire; some report overload and request clearer “starter” frameworks. |

### 4️⃣ Hands‑on / Experiential Learning  

| Condition | Action / Interaction | Consequence |
|-----------|----------------------|-------------|
| **In‑class data collection (transcribing, interviewing, observation)** (Chunks 6, 15, 31, 101, 122) | Conduct mock interviews, record field notes, transcribe instructor‑provided transcript | Students report *confidence* in data handling, but time constraints limit depth of reflection. |
| **Group coding / peer feedback** (Chunks 15, 18, 42, 67, 135, 166) | Small‑group coding sheets, think‑pair‑share, jamboard, peer‑review of themes | Enhanced sense of community, richer code diversity; occasional “groupthink” when dominant voices dominate. |
| **Use of visual / artistic media (films, documentaries, art‑based assignments)** (Chunks 22‑24, 31, 70‑74, 112‑115) | Show films, ask students to code scenes, create found‑poetry, discuss emotional impact | Strong affective engagement, deeper empathy; some view the activity as “extra workload”. |
| **Iterative practice (multiple coding passes, re‑analysis)** (Chunks 76, 131, 172, 174) | Encourage revisiting data with new lenses, re‑coding after discussion | Students develop reflexivity, but may feel “never‑ending” without clear endpoint. |
| **Physical tools (highlighters, coloured pens, printed sheets)** (Chunks 20‑23, 41, 198) | Provide colour‑coded worksheets, use columns/rows for themes vs feelings | Visual organization aids memory; reinforces manual preference. |

### 6️⃣ Assessment, Feedback & Evaluation  

| Condition | Action / Interaction | Consequence |
|-----------|----------------------|-------------|
| **Process‑oriented grading (reflection, memo, theme list)** (Chunks 29, 45, 204, 205) | Grade on coding rationale, memo quality, reflective writing rather than final product | Students focus on *thinking* rather than “getting the right answer”; some miss concrete performance metrics. |
| **Lack of clear rubric / expectations** (Chunks 30, 44, 140, 205) | Provide vague “framework” or “4Ws” without detailed scoring guide | Confusion, perceived unfairness, but also encourages autonomy for some learners. |
| **Peer‑review and collaborative evaluation** (Chunks 135, 160, 207) | Use Jamboard, group presentations, peer‑feedback forms | Improves critical thinking, builds community; occasional anxiety about peer judgment. |
| **Instructor‑driven “demo” coding** (Chunks 32, 42, 98, 198) | Live coding on screen, step‑by‑step walkthrough | Clarifies process, reduces uncertainty; students still report difficulty transferring to independent work. |

## How to read the map  

* **Condition** – the trigger or context that participants repeatedly mentioned (e.g., “lack of prior qualitative exposure”, “tight semester schedule”).  
* **Action / Interaction** – what the instructor or student did in response (e.g., “provide colour‑coded worksheets”, “run a group‑coding exercise”).  
* **Consequence** – the observed outcome, both affective (confidence, frustration) and cognitive (better understanding, continued uncertainty).  

The map shows that **most consequences are ambivalent**: the same action (e.g., hands‑on coding) can produce *positive* learning gains *and* generate *new uncertainties* that later require additional scaffolding.  This tension is the core dynamic that shapes how qualitative methods are taught and learned in the program.
